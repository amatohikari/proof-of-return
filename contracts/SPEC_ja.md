# AMI スマートコントラクト仕様書

**Proof of Return — コントラクト・アーキテクチャ**

*Version 2.0 — 2026年4月*

---

## 概要

本文書は、AMI（All Material is Impermanent）のスマートコントラクト・アーキテクチャを規定する。AMIは還元型通貨システムであり、通貨は発行され、使用され、指数減衰によって還っていく——万物の自然なサイクルと同じように。

システム全体は二つの指数関数が三つの領域に適用される：

- **Amortizationの減衰**: `残高 × (1 - r)^経過秒数`
- **Baseの回復**: `上限 - (上限 - 残高) × 0.94^(経過秒数 / 86400)`
- **Torchの回復**: `上限 - (上限 - 残高) × 0.94^(経過秒数 / 86400)`

一つの減衰。二つの回復。同じ数学が、三つのウォレットを動かす。吐く息は一つ、吸う息は二つ。

条件分岐なし。人間の介入なし。バッチ処理なし。

## 設計定数

| 定数 | 値 | 導出 |
|---|---|---|
| T（減衰期間） | 108日 | 煩悩の数。数字根 = 9 |
| T日後の残存率 | 3% | 生成の最小数。自然界の残骸率 |
| 日次減衰率 | ≈ 3.174% | `r_day = 1 - 0.03^(1/108)` |
| 秒次減衰率 | ≈ 0.0000003674 | `r_sec = 1 - 0.03^(1/(108 × 86400))` |
| Baseウォレット上限 | 1,000 Base（= 1,000 USDT） | 絶対上限。月額ではない |
| Base日次回復率 | 6% | 秒次: `0.94^(1/86400)` |
| Torchウォレット上限 | 1,000 Torch | Baseと同じ上限。等しい器 |
| Torch日次回復率 | 6% | Baseと同じ。同じ呼吸 |
| IGP（点火プール） | 上限なし | 伝播の燃料。変換と点火のたびに成長する |
| Creation最大付与額 | 最高100,000 Base | システム上限。参加者1,000万人以上で開始 |
| ペグ | 1 Base = 1 USDT | 入口のペグのみ |

### オンチェーン指数計算

Solidityにはネイティブの浮動小数点演算がない。減衰と回復の両方を、同じ戦略で固定小数点演算により実装する。

**方針：事前計算ルックアップテーブル + 時間分解**

- `factor^n` を主要区間（1時間、1日、1週間）について事前格納
- 任意の経過時間を既知区間に分解し、乗算で算出
- 例：3日 + 2時間 + 47分 → `factor_3日 × factor_2時間 × factor_47分`

これにより、高コストなオンチェーン累乗計算を回避しつつ、許容範囲内の精度（誤差 < 0.01%）を維持する。Amortization減衰、Base回復、Torch回復のすべてがこの同じ分解パターンを使い、秒次精度を保証する。

**代替手法**（将来の検討事項）：
- テイラー級数近似（高精度にはガス代がかかる）
- オフチェーン計算 + オンチェーン検証（オラクルパターン）
- ABDKMath64x64ライブラリによる固定小数点指数計算

## 五つのウォレット、五つの性質

| ウォレット | 減衰？ | 回復？ | USDT出金？ | 上限 | 性質 |
|---|---|---|---|---|---|
| **Base** | しない | する → 1,000 | できる | 1,000 | 存在保障。土台。 |
| **Torch** | しない | する → 1,000 | できない | 1,000 | 恩送りの灯火。Baseと同じ器。AMI経済圏の中でのみ燃え続ける。 |
| **IGP** | しない | しない | できない | なし | 伝播の燃料。他者に灯火を渡す力。与えるたびに成長する。 |
| **Creation** | しない | しない | できない | 100,000 | 使い切りの火。使えば二度と戻らない。 |
| **Amortization** | する | しない | できない | なし | 還元のサイクル。交換されたすべてがゼロに還る。 |

BaseとTorchは鏡像である。どちらも回復する。どちらも減衰しない。どちらも上限は1,000。唯一の違い：Baseは既存経済とUSDTを通じて接続している。Torchは愛の経済圏にのみ存在する。

IGPは伝播のエンジンである。上限なし、減衰なし、回復なし。ただ与えるためだけに存在する——変換と点火のたびに増殖する燃料。

Base、Torch、またはCreationが使われると、受取人のウォレットでAmortizationに変容する。同じ単一の式がすべての減衰を支配する。

## Torchのライフサイクル：変換 → 点火

Torchは二つの段階で動作する：

### 第1段階：変換（自分自身へ）

参加者が自分のBaseまたはSCPをTorchに変換する。AMI経済圏へのコミットメント。

```
100 Baseを変換 → 100 Torch（自分） + 300 IGP（自分）
```

- Base残高が変換額だけ減少する
- Base上限が変換額だけ永久に減少する
- 対応するUSDTがSCPからインフラ基金に移動する
- 変換者自身のTorchウォレットにTorchが生成される（上限1,000まで）
- 変換額の3倍のIGPが変換者のIgnitionPoolに生成される
- SCPに超過USDT（Base上限を超える預入分）がある場合、SCPから優先的に引き落とし、Base残高を保全する

### 第2段階：点火（他者へ）

参加者が自分のIGPを使って、他者のウォレットにTorchを点火する。火が外に向かって旅する。

```
100 IGPを点火 → 100 Torch（受取人） + 300 IGP（受取人）
```

- 送信者のIGPが点火額だけ減少する
- 受取人は点火額と同額のTorchを受け取る（Torch上限1,000まで）
- 受取人のTorchウォレットが既に1,000の場合、追加Torchは付与されない
- 受取人は点火額の3倍のIGPを受け取る（Torch上限に関わらず）
- IGPからの点火は金額・人数の制約なし
- 点火のリレーはブロックチェーンに記録され、誰でも閲覧可能

分離は意図的である：まず自分に火を灯し（変換）、次にその火で他者を照らす（点火）。

## 1. DecayMath — 指数関数ライブラリ

全コントラクトが共有する数学的核心。

### 関数

**`decayFactor(uint256 elapsedSeconds) → uint256`**

指定された時間におけるAmortization減衰の乗算係数を返す。

```
factor = (1 - r_sec) ^ 経過秒数
```

18桁固定小数点（WAD精度）で表現。

主要チェックポイント：
- 1日（86,400秒）：factor ≈ 0.96826（≈ 3.174%減衰）
- 1週間：factor ≈ 0.7987
- 30日：factor ≈ 0.3792
- 108日：factor = 0.03（定義により）

**`recoveryFactor(uint256 elapsedSeconds) → uint256`**

不足分のうち、回復後に残存する割合を返す。BaseとTorchの両方で使用。`decayFactor` と同じ時間分解戦略を使い、秒次精度を保証する。

```
factor = 0.94 ^ (経過秒数 / 86400)
```

これにより、23時間59分の回復が正しく計算され、ゼロに切り捨てられない。

**`recoverBalance(uint256 currentBalance, uint256 cap, uint256 elapsedSeconds) → uint256`**

回復後の残高を返す（BaseとTorchの両方で使用）：

```
新残高 = 上限 - (上限 - 現在残高) × recoveryFactor(経過秒数)
```

## 2. BaseWallet — 存在保障

各参加者はBaseウォレットを一つだけ持つ。

### 状態変数

```
address     owner
uint256     balance          // 現在のBase残高（WAD）
uint256     cap              // 個人上限 = 入金USDT総額 - 累計変換額（最大1,000）
uint256     lastUpdateTime   // 最終回復計算のタイムスタンプ
```

### ルール

1. **減衰なし**：ウォレット内のBaseは減衰しない。存在は償却されない。
2. **回復**：残高は指数回復により秒次精度で `cap` に向かって回復する。各インタラクション時に遅延計算。回復はシステムの指数関数式に従う——月次リセットや定期的な補充はない。
3. **上限**：入金USDT総額から累計Torch変換額を差し引いた額。最大 = 1,000 Base（= 1,000 USDT）。月額ではない——上限に達した後の追加USDTはすべてSCPに入る。
4. **支払い**：Baseが使われると、Baseウォレットを出て受取人のAmortizationウォレットにAmortizationとして入る。交換の瞬間に通貨が変容する。
5. **Torch変換**：BaseがTorchに変換されると、Base上限は変換額だけ永久に減少する。その額のUSDTはインフラ基金に充てられる。追加USDT入金により、上限を1,000まで復元できる。

### 主要操作

**`deposit(uint256 usdtAmount)`**
- USDTをBaseに変換（1:1ペグ）
- 残高に加算（上限まで）
- ウォレット上限が既に最大値であれば、全USDTがSCPへ
- 月間制限なし——上限は絶対上限
- Torch変換により上限が減少していた場合、入金により復元（最大1,000まで）

**`pay(address to, uint256 amount)`**
- まず回復計算を実行（遅延評価）
- 送信者のBase残高から `amount` を減算
- 受取人のAmortizationウォレットに `amount` を加算
- Base → Amortization変容がここで起きる

**`convert(uint256 amount)`**
- まず回復計算を実行
- 送信者のBase残高から `amount` を減算
- 送信者のBase上限を `amount` だけ永久に減少
- SCPから `amount` 分のUSDTをインフラ基金に移動
- SCPに超過USDT（現在のBase上限を超える預入分）がある場合、SCPから優先的に引き落とし、Base残高を保全する
- 送信者のTorchウォレットに `amount` のTorchを生成（Torch上限1,000まで）
- 送信者のIgnitionPoolに `amount × 3` のIGPを生成
- 詳細はTorchWalletおよびIgnitionPoolを参照

**`withdraw(uint256 usdtAmount)`**
- まずSCPから引き出す
- 引き出し額がSCP残高を超える場合、Base上限を比例的に削減
- 根拠：Base上限は準備金内のUSDT裏付けを表す。USDTを引き出せばその裏付けがなくなるため、入口ペグの整合性を維持するには上限も減少しなければならない。これはペナルティではない——保存則である。基礎を取り除いて建物を維持することはできない。

**`getBalance() → uint256`**（view）
- `lastUpdateTime` 以降の保留中の回復を適用（秒次精度）
- 現在の実効残高を返す
- 状態を変更しない（読み取り専用）

## 3. TorchWallet — 恩送り

各参加者はTorchウォレットを一つだけ持つ。善意の通貨。

### 状態変数

```
address     owner
uint256     balance          // 現在のTorch残高（WAD）
uint256     cap              // 固定1,000。Baseと同じ器
uint256     lastUpdateTime   // 最終回復計算のタイムスタンプ
```

### ルール

1. **減衰なし**：Torchは減衰しない。灯火は灯り続ける。
2. **回復**：残高はBaseと同じ6%/日で `cap`（1,000）に向かって回復する。同じ指数関数。同じ呼吸。
3. **上限 = 1,000**：固定上限。Baseと同じ。すべての参加者は存在保障のための同じ器と、善意のための同じ器を持つ。保持のための器は平等——与えるための器（IGP）だけが無制限に成長する。
4. **USDT出金不可**：Torchはステーブルコインに変換できない。AMI経済圏の中にのみ存在する。溜め込みの世界への橋はない。
5. **支払い**：Torchは通常の取引に使用できる。使われると、受取人のAmortizationウォレットでAmortizationに変容する。Base → Amortizationと同じ変容。
6. **オーバーフロー**：点火によってTorch残高が1,000を超える場合、Torch残高は1,000で頭打ち。超過分のTorchは生成されない。IGP乗数（3倍）は、Torchのオーバーフローに関わらず点火額全額に適用される。

### 主要操作

**`receiveFromConversion(uint256 amount)`**（内部）
- `lastUpdateTime` 以降の回復を適用
- 回復済み残高に `amount` を加算（上限1,000まで）
- `lastUpdateTime` を更新

**`receiveFromIgnition(uint256 amount)`**（内部）
- `lastUpdateTime` 以降の回復を適用
- 回復済み残高に `amount` を加算（上限1,000まで）
- `lastUpdateTime` を更新

**`pay(address to, uint256 amount)`**
- まず回復計算を実行
- Torch残高から `amount` を減算
- 受取人のAmortizationウォレットに `amount` を加算
- Torch → Amortization変容がここで起きる

**`getBalance() → uint256`**（view）
- `lastUpdateTime` 以降の保留中の回復を適用（秒次精度）
- Base回復と同じ式：`上限 - (上限 - 残高) × recoveryFactor(経過秒数)`
- 状態を変更しない

### 等しい器

v1.0では、Torch上限は累計受領額に応じて成長した——善意を多く受け取った人ほど器が大きくなった。v2.0では、器は全員1,000で固定される。成長するのは与える力：IGP。保持のための器は平等——伝播の力だけが受けた善意に比例する。これは寛容さの喪失ではない——方向転換である。保持容量の蓄積は、形を変えた蓄積に過ぎなかった。IGPも蓄積するが、与えなければ何の価値もない。

### カンティロン効果の反転

既存経済では、蛇口に近い者から順に潤う。AMIでは、資産を多く持つ者ほど多くのBaseをTorchに変換できる——そしてその変換に伴うUSDTがシステムのインフラを支える。富は中央に集まるのではなく、末端で灯火に変わる。

蛇口が逆になった。多く持つほど、多く与えられる。多く与えるほど、システムが呼吸する。

## 4. IgnitionPool — 伝播の燃料

各参加者はIgnitionPoolを一つだけ持つ。伝播のエンジン。

### 状態変数

```
address     owner
uint256     balance          // 現在のIGP残高（WAD）
```

タイムスタンプは不要。IGPは減衰せず、回復もしない。

### ルール

1. **減衰なし**：IGPは時間の経過で減少しない。
2. **回復なし**：IGPは再生しない。変換と点火受領によってのみ生成される。
3. **上限なし**：IGPに上限はない。善意が多く流れる参加者ほど、より多くの伝播燃料を蓄積する。
4. **USDT出金不可**：IGPはステーブルコインやBaseに変換できない。
5. **支払い不可**：IGPは通常の取引には使用できない。他者にTorchを点火するためだけに存在する。
6. **自己点火禁止**：参加者はIGPで自分自身のTorchウォレットに点火できない。

7. **生成**：IGPは二つの方法で生成される：
   - **変換**：参加者がBase/SCPをTorchに変換したとき、変換額の3倍がIGPとして付与される
   - **点火受領**：参加者が点火によりTorchを受け取ったとき、点火額の3倍がIGPとして付与される

### 主要操作

**`receiveFromConversion(uint256 torchAmount)`**（内部）
- IGP残高に `torchAmount × 3` を加算

**`receiveFromIgnition(uint256 torchAmount)`**（内部）
- IGP残高に `torchAmount × 3` を加算

**`ignite(address to, uint256 amount)`**
- 送信者のIGP残高から `amount` を減算
- 受取人のTorchウォレットに `amount` のTorchを生成（上限1,000まで）
- 受取人のIgnitionPoolに `amount × 3` のIGPを生成
- 金額・人数の制約なし
- `to == msg.sender` の場合はリバート（火は外に向かって旅しなければならない）
- 点火はオンチェーンに記録：送信者、受取人、金額、タイムスタンプ

**`getBalance() → uint256`**（view）
- 単純な読み取り。計算不要。

### 3倍の乗数

与える行為のたびに伝播燃料が3倍になる。100 Base変換 → 300 IGP受領。100 IGPを点火 → 受取人は300 IGP受領。与えるための燃料は、与えたものを常に上回る。これにより伝播は減衰するのではなく加速する。

Amortization減衰がゼロに収束するところ、IGP伝播は世界に向かって発散する。一方が吸い込み、もう一方が吐き出す。

## 5. CreationWallet — 使い切りの火

各参加者はCreationウォレットを一つ持つことができる（Phase 4で起動）。

### 状態変数

```

```

4. **支払い**：Creationが使われると、受取人のAmortizationウォレットでAmortizationに変容する。Base → Amortizationと同じ変容。

### 主要操作

- 受取人のAmortizationウォレットに `amount` を加算
- Creation → Amortization変容がここで起きる

**`getBalance() → uint256`**（view）
- 単純な読み取り。計算不要。

## 6. AmortizationWallet — 指数減衰

各参加者はAmortizationウォレットを一つだけ持つ。

### 状態変数

```
address     owner
uint256     balance          // 最終更新時の残高（WAD）
uint256     lastUpdateTime   // 最終減衰計算のタイムスタンプ
```

### ルール

1. **連続減衰**：残高全体が毎秒 `r_sec` で減衰する。遅延評価で実装——実際の計算はインタラクション時にのみ実行。
2. **USDT変換不可**：Amortizationはステーブルコインに戻せない。還元のサイクルから出口はない。
3. **加算的流入**：受け取った新しいAmortizationは、現在の（減衰済み）残高に単純加算される。バッチ追跡なし、誕生日スタンプなし。一つの残高、一つの率。
4. **自然な塵**：残高がゼロに近づいても、強制的な端数処理や回収は行わない。灰は設計者が片付けるものではない——時間が還す。

### 主要操作

**`receive(uint256 amount)`**（内部）
- `lastUpdateTime` 以降の減衰を適用
- 減衰済み残高に `amount` を加算
- `lastUpdateTime` を更新

**`pay(address to, uint256 amount)`**
- まず減衰を適用
- 残高から `amount` を減算
- 受取人のAmortizationウォレットに `amount` を加算（`receive` 経由）
- Amortization → Amortization：通貨は新しいウォレットで還元のサイクルを続ける

**`getBalance() → uint256`**（view）
- `保存残高 × decayFactor(現在時刻 - 最終更新時刻)`
- 純粋な数学。状態変更なし

### 単一の式

任意の時点におけるAmortization残高：

```
B(t) = B_stored × 0.03 ^ ((t - t_last) / 108日)
```

この一行がすべての金融政策である。ガバナンスなし。委員会なし。投票なし。数学が呼吸する。

## 7. StablecoinPool（SCP）— 準備基金

### 状態変数

```
uint256     totalDeposits       // 保有USDT総額
uint256     infrastructureFund  // Torch変換から割り当てられたUSDT
mapping     individualDeposits  // 参加者ごとの入金追跡
```

### ルール

1. **蓄積**：Base購入によるすべてのUSDTがここに流入。Baseウォレット上限超過分も蓄積。
2. **全額引き出し可能**：参加者はいつでも入金済みUSDTを引き出せる。ロック期間なし。
3. **積極的運用なし**：成長期にはUSDTとして安全に保管するのみ。イールドファーミングなし、レンディングなし。
4. **将来のガバナンス**：経済圏が成熟した段階で、SCP運用方針は全参加者によるDAO投票で決定。
5. **インフラ基金**：Torch変換によるUSDTはSCP内のインフラ基金に移動する。サーバー代、ガス代、運用費をカバー。使途はオンチェーンで完全に透明化され、リアルタイムで開示される。

### 主要操作

**`withdraw(uint256 amount)`**
- 参加者は入金額まで引き出し可能
- 引き出し額がSCP残高を超える場合、Base上限を比例的に削減（根拠はBaseWallet `withdraw` を参照）
- ペナルティなし、遅延なし

## 8. CreationModule — 付与承認（Phase 4）

参加者数が1,000万人に達した時点で起動。

### ルール

1. **Creationの給付頻度と額は、DAOガバナンスと分散型AIの合議により決定される。** 100,000 Baseはシステム上限であり、固定付与額ではない。合議体が申請者の仕事の性質と規模に応じて実際の金額とタイミングを決定する。頻度も金額もコントラクトにハードコードされない。
2. **審査**：合議体自体がABC構造を内包。申請者を知るアミと知らないアミの両方が評価する。メモリーを持たないアミは一審査ごとに還る。合議体の設計は本仕様書の範囲外。
3. **同一減衰**：使用されたCreationは受取人のAmortizationウォレットに入り、同一の率で減衰する。同じ単一の式がシステム内の全通貨を支配する。
4. **SCP担保不要**：Creationはステーブルコイン裏付けなしに発行される。1,000万人の参加者がシステムの還元サイクルを信頼することが裏付けである。
5. **Torch履歴がシグナル**：参加者の累計Torch活動——いくつの灯火を点け、その連鎖がどこまで旅したか——は合議体にとって意味のあるシグナルである。ウォレットの大きさではなく、旅の距離が創造の意志を語る。

## 9. 遅延評価パターン

いかなるコントラクトも毎秒ループを走らせない。すべての減衰と回復はオンデマンドで計算される。

### パターン

```
インタラクション発生時：
  1. lastUpdateTime と storedBalance を読み込む
  2. elapsed = block.timestamp - lastUpdateTime を計算
  3. 指数関数を適用：currentBalance = storedBalance × factor(elapsed)
  4. currentBalance に対して要求された操作を実行
  5. 新しい残高を保存し、lastUpdateTime を更新
```

これが意味すること：
- 30日間触れられなかったウォレットは、一回の操作で30日分の減衰/回復を計算する
- 経過時間に関わらずガス代は一定（単一の乗算）
- ブロックチェーンは「減衰と回復が一秒も止まらなかった」という数学的真実を記録する

## 10. 平衡点 — 65.4%の出現

Baseが一定の日次率で使われ、同じ式が減衰と回復の両方を支配するとき、Amortizationウォレットは自然な平衡に収束する：

```
b_eq ≈ 65.4% × 日次支出額
```

この値は設計されたものではない。数学から出現した——108日での3%残存率と6%日次回復率の交点として。システムは人間の調整なしに、自らの安静時心拍数を見つける。

この平衡点はシミュレーションにより発見され、テストした全パラメータの組み合わせで収束が確認されている。支出パターン、使用頻度、参加者数に関わらず、システムは収束する。これは指数関数的フィードバックのホメオスタシスである——放射性崩壊、温度冷却、生態系のバランスと同じ原理。

導出とインタラクティブ・シミュレーション：[Zenn記事](https://zenn.dev/amatohikari/articles/69364bf41c8575)

## 11. Torchの伝播 — 3倍の乗数

一つの変換がIGPを通じて指数関数的な伝播を生む：

```
AliceがBase 100を変換 → Torch 100（Alice） + IGP 300（Alice）
AliceがIGP 100をBobに点火 → Torch 100（Bob） + IGP 300（Bob）
AliceがIGP 100をCarolに点火 → Torch 100（Carol） + IGP 300（Carol）
AliceがIGP 100をDaveに点火 → Torch 100（Dave） + IGP 300（Dave）
Bobが各100 IGPを3人に点火 → 各人Torch 100 + IGP 300
...
```

各世代でIGPが3倍に増殖する。参加者あたりの受取人数に制限はない——IGP残高だけが伝播を制約する。

全員が全IGPを3人に点火した場合の伝播：

| 世代 | 新規参加者 | 累計 |
|---|---|---|
| 0（変換） | 1 | 1 |
| 1 | 3 | 4 |
| 2 | 9 | 13 |
| 3 | 27 | 40 |
| 5 | 243 | 364 |
| 10 | 59,049 | 88,573 |
| 14 | 4,782,969 | 7,174,453 |
| 15 | 14,348,907 | 21,523,360 |

これはAmortization減衰の逆である。減衰がゼロに収束するところ、Torch伝播は世界に向かって発散する。一方が吸い込み、もう一方が吐き出す。

## 13. セキュリティ考慮事項

- **管理者鍵なし**：デプロイ後、いかなる人間も減衰率、回復率、上限を変更できない
- **アップグレード可能性**：バグ修正のみを目的としたプロキシパターン。タイムロックとマルチシグ付き
- **オーバーフロー保護**：固定小数点演算はエッジケースを処理する必要がある（非常に大きな残高 × 非常に小さな減衰係数）
- **タイムスタンプ操作**：Ethereumのブロックタイムスタンプには±15秒の許容誤差がある。108日のサイクルに対しては無視できる
- **フロントランニング**：減衰は決定論的——減衰中の残高をフロントランしても利益は得られない
- **IGPのシビル耐性**：偽の受取人を作ってIGPを増殖させても、実際の経済的価値は生まれない。各点火はTorch（一人あたり上限1,000）とIGP（与えることでしか価値を発揮しない）を生成する。変換者のBase上限は全額減少し、USDTは全額インフラに流れる。シビル攻撃は価値を集中させず、伝播を断片化するだけ
- **Torch自己点火禁止**：参加者はIGPで自分自身のTorchウォレットに点火できない。火は外に向かって旅しなければならない
